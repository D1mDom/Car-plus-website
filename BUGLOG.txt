BUG #001
DATE FOUND: 7/22/2026
FOUND BY: Nimith (with Claude)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- The admin panel and admin nav link were visible to everyone, and the guards
  in Admin / AdminOrders / AdminReports never actually blocked anyone.

CAUSE (root cause):
- src/hooks/useAdmin.tsx was hardcoded to return { isAdmin: true }. The real
  admin query was written but unreachable (comment: "Bypassed for local
  testing"). Every "if (!isAdmin)" check was therefore dead code.
- The database RLS still blocked non-admin writes, so this was a UI/exposure
  bug, not a data-theft bug — but strangers could open the admin UI.

HOW IT WAS FIXED:
- Replaced the stub with a real call to the is_admin RPC, using the signed-in
  user's id. Returns false when signed out or when the check errors (fail
  closed), so a failure never grants access.

FILES CHANGED:
- src/hooks/useAdmin.tsx

BUG #002
DATE FOUND: 7/22/2026
FOUND BY: Nimith (with Claude)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- After completing checkout the customer sees a success screen, but the order
  never appears in /orders.

CAUSE (root cause):
- src/pages/Checkout.tsx handleSubmit is a mock: it waits with setTimeout, then
  writes the order to localStorage ('order_history'). It never inserts into the
  Supabase orders / order_items tables. But src/pages/Orders.tsx reads orders
  from Supabase, so the two never meet. The cart (useCart) is also localStorage
  only, while a cart_items table with RLS sits unused.

HOW IT WAS FIXED:
- handleSubmit now inserts into orders, then order_items (using the returned
  order id), then clears the cart. Removed the setTimeout mock and the
  localStorage order_history write.

FILES CHANGED:
- src/pages/Checkout.tsx

BUG #003
DATE FOUND: 7/22/2026
FOUND BY: Nimith (with Claude)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- Car photos had to be very small or the site got slow; large uploads were
  impractical.

CAUSE (root cause):
- The admin form converted each image to a base64 data URL and stored the whole
  string in cars.image. base64 inflates size ~33%, the image lived inside the
  DB row, and useCars does select("*"), so every page load downloaded every
  car's full image inside one JSON response. No CDN caching, no thumbnails.

HOW IT WAS FIXED:
- Rewrote CarFormDialog to upload the file to the car-images storage bucket and
  store only the returned public URL. Added client-side compression (max
  1920px, WebP at 0.85), so even a 50 MB original becomes a few hundred KB.
- Added the storage bucket + policies to supabase_setup.sql.

FILES CHANGED:
- src/components/admin/CarFormDialog.tsx
- supabase_setup.sql

NOTE: Not yet tested against a live database (none existed). First upload after
the new project is set up is the real test.

BUG #011
DATE FOUND: 7/22/2026
FOUND BY: Nimith (with Claude)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- Setting an order to "shipped" or "delivered" from the admin UI would be
  rejected by the database.

CAUSE (root cause):
- The orders.status CHECK constraint in supabase_setup.sql allowed only
  pending/confirmed/processing/completed/cancelled, but useAdminOrders'
  ORDER_STATUSES offers shipped and delivered too. Three different status
  vocabularies existed across the SQL, the admin hook, and Orders.tsx.

HOW IT WAS FIXED:
- Widened the CHECK constraint (both the CREATE TABLE and the section-6 ALTER)
  to the union: pending, confirmed, processing, shipped, delivered, completed,
  cancelled.

FILES CHANGED:
- supabase_setup.sql

BUG #012
DATE FOUND: 7/22/2026
FOUND BY: Nimith (with Claude)
STATUS: FIXED
-------------------------------------------
WHAT WAS WRONG:
- A saved cart / wishlist could be wiped on page reload.

CAUSE (root cause):
- In useCart and useWishlist the "save to localStorage" effect was guarded only
  by !loading, so on first mount (items = [], loading = false) it could write an
  empty array to storage before the initial fetch read it back.

HOW IT WAS FIXED:
- Added a `hydrated` flag set true only after the first fetch runs; the save
  effect now waits for `hydrated`, so an empty initial state can't overwrite
  stored data.

FILES CHANGED:
- src/hooks/useCart.tsx
- src/hooks/useWishlist.tsx

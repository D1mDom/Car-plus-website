DONE TODAY:
- Font: switched headings from Bokor to Koulen, then applied Koulen site-wide
  (index.html font link, tailwind.config.ts, src/index.css). Fallbacks kept
  (Battambang, Inter) behind it.

  Inter font. Moved it to the top.
- Added vercel.json with an SPA rewrite so routes like /admin and /wishlist do
  not 404 on refresh when deployed.
- Added .vercel to .gitignore.
- Rewrote supabase_setup.sql into a full, idempotent setup script:
  - Storage bucket "car-images" (public read, admin-only write, 50 MB limit,
    image MIME types only).
  - 10 indexes on the columns the app actually queries (were zero before).
  - CHECK constraints (price/total >= 0, quantity > 0, orders.status whitelist).
  - Missing foreign keys added: wishlist.user_id and admin_users.user_id now
    reference auth.users with ON DELETE CASCADE.
  - Unique constraints: cart_items(user_id, car_id), wishlist(user_id, car_id).
  - Bootstrap instructions for granting the first admin.
  - Section 6 "upgrade" block so an existing DB can be brought up to date.
- Rewrote src/components/admin/CarFormDialog.tsx image handling: was saving the
  photo as a base64 data URL into the cars.image column. Now uploads the file to
  the car-images storage bucket and stores only the public URL. Added
  client-side compression (scale to max 1920px, re-encode to WebP at 0.85) so a
  large phone/DSLR photo becomes a few hundred KB before upload. 50 MB input cap.
- Fixed src/hooks/useAdmin.tsx: it was hardcoded to return isAdmin: true for
  everyone. Now calls the is_admin RPC and fails closed (false) when signed out
  or on error. Committed on branch storage-image-upload as a PR.

DECISIONS MADE:
- car_id columns stay TEXT (not a UUID foreign key to cars). The app falls back
  to the static catalogue in src/data/cars.ts whose ids are plain strings
  ("1","2",...). A FK would break that fallback.
- Upload limit set to 50 MB but paired with WebP compression, so stored files
  stay small. This also fixes the "uploads are strict / must be tiny" problem,
  which was really caused by the base64 approach, not a Supabase limit.
- Stay on Supabase. Not switching backends and not building custom auth: the app
  is coupled to Supabase auth + RLS across 10 files, 3 edge functions, and all
  the security policies. Switching would be a full rewrite, wrong for a demo.
- Koulen applied site-wide at Nimith's request. Noted trade-off: Koulen has a
  single weight, so bold/semibold no longer look heavier than normal text.
- For the demo: recommended creating a NEW free Supabase cloud project (fast, no
  install, keeps all existing code) over local Supabase (needs Docker, not
  installed) or a backend switch.
